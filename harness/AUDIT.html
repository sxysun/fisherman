Fisherman Harness Architecture and Gap Audit

A direct assessment of the proactive-presence harness as of May 22, 2026: what exists, why the architecture mostly makes sense, where it falls short of a frontier-lab-quality system, and what has to close before it is a reliable working harness rather than a promising dogfood prototype.

Scope: harness/ plus app boundary checks Smoke tests: 56 passing Approx. source/docs: 55 files, 10k lines Updated: 2026-05-22

Executive Verdict

The harness is architecturally coherent for a local research prototype. It has the right primitive loop: observe screen state, turn it into a candidate event, decide whether to ping, realize a goal-aware message, push it through a low-friction UI, capture outcomes, and audit policy behavior offline.

It is now a serious local dogfood harness, but not yet a validated policy. The engineering gaps called out in the first audit now have concrete implementations: deterministic holdouts, endpoint trust boundaries, native metrics, durable trace lifecycle, workflow-event joins, KG priors, hard-example mining, curation records, launchd packaging, capsule respawn, masked settings keys, all-Spaces capsule behavior, and SQLite-backed read paths. The remaining gap is evidence: enough labels, outcomes, and frozen evaluation slices to justify policy changes.

Best current use: dogfood on one machine with an explicit daily goal, collect labels, and use the trace data to harden the policy. Do not treat current ping quality as validated.

Repository Boundary

Commit model

harness/ should be committed in this repository because it depends on Fisherman's local HTTP API and DynamicNotchKit. It should still be treated as a separate package and research harness, not as Fisherman app runtime.

  • Own Python package: harness/pyproject.toml
  • Own config/data root: ~/.harness
  • Own Swift notch app: harness/notch
  • Own CI workflow: .github/workflows/harness-ci.yml

Update isolation

The main app update checker is path-scoped and deliberately excludes harness/. Harness-only commits should not show up as "Update Fisherman" in the Settings -> Updates tab.

  • fisherman/cli.py uses _UPDATE_RELEVANT_PATHS; harness/ is not included.
  • The DMG payload list excludes harness/.
  • The app bootstrap/upgrade sync lists exclude harness/.
  • Main CI now ignores harness-only changes; Harness CI owns them.

How The Harness Works

1. Fisherman captureFisherman daemon exposes frames, OCR, app/window metadata, status, and query endpoints on localhost.
2. Candidate synthesiscandidate.py snapshots screen context and current user prefs into a typed CandidateEvent.
3. Scene readingRules tag coarse scenes. Optional VLM overlay adds specificity, sensitivity, and intent signals.
4. Short memorySessionMemory tracks recent apps/scenes, switches, time on current app, and recent outcomes.
5. Policyrule_v0 applies hard gates and fallback behavior; the default llm_icl_v0 learner chooses ping/not-ping from context plus labeled examples.
6. RealizerOpenAI-compatible chat endpoint receives daily goal, why-now, compact state, and optionally the current JPEG.
7. CriticRegex critic blocks obvious secrets, long quotes, verbose output, or empty/multi-paragraph messages. LLM critic is optional.
8. Capsule + rewardSwift capsule displays the ping, captures hover/approach/click/dismiss/timeout, and writes signal-derived reward.

Fisherman -> CandidateEvent -> Scene/VLM -> Memory -> Policy -> Realizer -> Critic -> Pending ping -> Outcome -> Reward -> Trace -> Labels/eval

What Is Strong

Right dependency direction

The harness depends on Fisherman through HTTP. Fisherman does not import or ship the harness as app runtime. This keeps the experimental loop replaceable.

Goal-aware intervention model

The current design avoids a brittle fixed-intent catalog. The gate emits why-now reasons and the realizer composes against the daily goal and actual visual context.

Feedback loop exists

Outcomes, hover traces, retro labels, implicit labels, score, and reward are in place. That is the right skeleton for improving intervention policy over time.

Native surface is appropriate

The floating capsule is less disruptive than a modal/chat UI, can coexist with Fisherman's notch, and can capture richer intent signals than a notification alone.

Local-first logs

All core trace artifacts live under ~/.harness. That makes dogfooding inspectable and easy to reset.

Testable seams

The modules are small enough to unit test: gate, scene overlay, reward, trace patching, and snooze behavior now have smoke coverage.

  • Made negative-feedback backoff time-bounded.
  • Made expired snoozes inactive instead of permanent blockers.
  • Made the "Later" notch action actually set a 30-minute snooze.
  • Made notch auto-dismiss respect the pending payload expiration.
  • Added local OCR privacy preflight before screenshot model calls.
  • Added harness-local screenshot masking for secret-like OCR boxes.

Data and eval

  • Preserved VLM specificity and intent signals through the event schema.
  • Included scene detail and intent signals in the realizer brief.
  • Attached outcomes and reward to matching trace rows.
  • Made hover-target reward target-aware, so hovering dismiss is soft rejection rather than positive consideration.
  • Added a privacy-safe model-call audit ledger for realizer, scene VLM, and LLM critic calls.
  • Added a typed SQLite sidecar so JSONL events can be queried without log scans.
  • Fixed the retro labeler queue so review sessions are frozen and skip moves forward instead of chasing live latest candidates.
  • Added live lab metrics for outcome capture, reward, label agreement, false interruptions, missed help, and data-readiness thresholds.
  • Added durable trace lifecycle patching so ping decisions get trace rows before realization starts.
  • Added workflow_event_id joins on candidates, decisions, and traces.
  • Added KG priors to the live llm_icl_v0 policy prompt.
  • Added hard-example mining, curation records, and frozen eval manifest export.
  • Added explicit precision/recall/F1 and trace-completeness metrics.

Risk Register

P0: Key rotation and screenshot privacy

A real-looking API key was present in the repo defaults before this audit. Even though source defaults are now blank, the old key remains in the local ~/.harness/config.toml until replaced. The harness now redacts secret-like OCR and masks sensitive screenshot text boxes locally before realizer model calls; if masking cannot be verified, screenshot attachment is suppressed.

P0: Metrics exist, but the sample is too small

The harness now reports false-interruption rate, missed-help rate, precision/recall/F1, trace completeness, outcome capture, reward, first-pass shadow-policy comparisons, hard-example counts, and deterministic holdouts. The remaining problem is data volume: a frontier lab would not trust these rates until there are enough labels and held-out slices.

P1: Storage has a typed sidecar, but not a migration contract

The harness now mirrors JSONL into SQLite and dashboard/metrics/replay/score/dataset builders/shadow prefer SQLite payload rows. It still needs explicit migration tests across old local databases, backfill/recovery checks, and retention policy before storage can be treated as long-lived lab infrastructure.

P1: Replay is still not the real system

Offline policy replay now simulates memory and cooldown better than before, but it still excludes realizer quality, critic behavior, VLM nondeterminism, user context, and delayed action. Treat it as a gate-regression tool, not a full offline product metric.

Milestone Concrete bar
Dogfood-ready Daemon starts via launchd, survives capsule restarts, keeps the capsule visible across Spaces, has no hardcoded secrets, has no update-check leakage, and the labeler clearly shows how many retro labels remain before personalization.
Policy-eval-ready At least 500 labeled candidates, frozen datasets, policy versioning, replay under 10 seconds for a day of candidates, and tracked precision, recall, F1, false-interruption rate, and missed-help rate.
Privacy-ready OCR preflight, line-box masking, endpoint allowlist, and model-call audit logs exist. Remaining bar: token-level masking, explicit per-provider policies, and sensitive-domain suppression.
Personalization-ready Few-shot policy examples from accepted/rejected local labels, learned or calibrated gate features, and rollbackable policy configs.
Production-grade SQLite-backed read paths, migrations, launchd packaging, observability, crash recovery, CI on all harness paths, release notes, security review, and documented rollback path.

Bottom Line

The core idea is sound: a proactive assistant should be evaluated as an intervention policy, not just as a chat prompt. This build has the right loop and enough instrumentation to start learning from real use.

The gap to "frontier lab" is disciplined measurement. The next useful work is labeled data, implicit-signal calibration, storage integrity, and policy evaluation.
