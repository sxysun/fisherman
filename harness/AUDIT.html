Fisherman Harness Architecture and Gap Audit

A direct assessment of the proactive-presence harness as of May 19, 2026: what exists, why the architecture mostly makes sense, where it falls short of a frontier-lab-quality system, and what has to close before it is a reliable working harness rather than a promising dogfood prototype.

Scope: harness/ plus app boundary checks Smoke tests: 28/28 passing Approx. source/docs: 55 files, 10k lines Generated: 2026-05-19

Executive Verdict

The harness is architecturally coherent for a local research prototype. It has the right primitive loop: observe screen state, turn it into a candidate event, gate interventions, realize a goal-aware message, push it through a low-friction UI, capture outcomes, and replay policy decisions offline.

It is not yet a frontier-lab-grade harness. The missing pieces are not mostly UI polish. They are counterfactual evaluation, privacy trust boundaries, experimentation discipline, migration/versioning discipline, and enough labeled data to tune or learn a policy. The first live metrics surface now exists, but it does not yet have enough labels behind it.

Best current use: dogfood on one machine with explicit Today goals, collect labels, and use the trace data to harden the policy. Do not treat current ping quality as validated.

Repository Boundary

Commit model

harness/ should be committed in this repository because it depends on Fisherman's local HTTP API and DynamicNotchKit. It should still be treated as a separate package and research harness, not as Fisherman app runtime.

  • Own Python package: harness/pyproject.toml
  • Own config/data root: ~/.harness
  • Own Swift notch app: harness/notch
  • Own CI workflow: .github/workflows/harness-ci.yml

Update isolation

The main app update checker is path-scoped and deliberately excludes harness/. Harness-only commits should not show up as "Update Fisherman" in the Settings -> Updates tab.

  • fisherman/cli.py uses _UPDATE_RELEVANT_PATHS; harness/ is not included.
  • The DMG payload list excludes harness/.
  • The app bootstrap/upgrade sync lists exclude harness/.
  • Main CI now ignores harness-only changes; Harness CI owns them.

How The Harness Works

1. Fisherman captureFisherman daemon exposes frames, OCR, app/window metadata, status, and query endpoints on localhost.
2. Candidate synthesiscandidate.py snapshots screen context and current user prefs into a typed CandidateEvent.
3. Scene readingRules tag coarse scenes. Optional VLM overlay adds specificity, sensitivity, and intent signals.
4. Short memorySessionMemory tracks recent apps/scenes, switches, time on current app, and recent outcomes.
5. Gaterule_v0 applies hard gates, cooldown, snooze, negative-feedback backoff, and goal/opportunity reasons.
6. RealizerOpenAI-compatible chat endpoint receives daily goal, why-now, compact state, and optionally the current JPEG.
7. CriticRegex critic blocks obvious secrets, long quotes, verbose output, or empty/multi-paragraph messages. LLM critic is optional.
8. Notch + rewardSwift notch displays the pill, captures hover/approach/click/dismiss/timeout, and writes signal-derived reward.

Fisherman -> CandidateEvent -> Scene/VLM -> Memory -> Gate -> Realizer -> Critic -> Pending pill -> Outcome -> Reward -> Trace -> Replay/labels

What Is Strong

Right dependency direction

The harness depends on Fisherman through HTTP. Fisherman does not import or ship the harness as app runtime. This keeps the experimental loop replaceable.

Goal-aware intervention model

The current design avoids a brittle fixed-intent catalog. The gate emits why-now reasons and the realizer composes against the daily goal and actual visual context.

Feedback loop exists

Outcomes, hover traces, retro labels, replay, score, and reward are in place. That is the right skeleton for improving intervention policy over time.

Native surface is appropriate

The notch pill is less disruptive than a modal/chat UI and can capture richer intent signals than a notification alone.

Local-first logs

All core trace artifacts live under ~/.harness. That makes dogfooding inspectable and easy to reset.

Testable seams

The modules are small enough to unit test: gate, scene overlay, reward, trace patching, and snooze behavior now have smoke coverage.

Critical Gaps vs. A Frontier-Lab Harness

Area Current state Frontier-lab bar Priority
Evaluation data Live traces, sparse retro labels, and a metrics endpoint/CLI now report ping rate, outcome capture, agreement, false-interruption rate, and missed-help rate. Gold datasets, human labels, disagreement analysis, held-out evals, and stable policy acceptance criteria. P0
Counterfactual policy eval Replay can rerun the gate over candidates, but it does not model realizer quality, user mood, delayed outcomes, or randomized holdout. Shadow policies, exploration buckets, calibrated propensities, IPS/doubly robust estimates where appropriate. P0
Privacy before network Local OCR preflight redacts secret-like text. Sensitive screenshots are locally masked using Apple Vision text boxes before realizer model calls; if masking fails, the image is suppressed. Token-level masking beyond whole text-line boxes, per-domain allow/deny policy, and explicit endpoint trust boundaries. P1
Persistent storage JSONL files under ~/.harness are mirrored into harness.db with typed tables for candidates, decisions, traces, outcomes, labels, and model calls. A CLI backfill can rebuild the sidecar from JSONL. Pending notch delivery is leased and completed on outcome. Formal migrations, read-path migration, retention/compaction, and automated recovery/backfill validation. P1
Policy learning Handwritten rule_v0 with simple sensitivity settings. Label-informed personalization, calibrated learned gate, exploration/exploitation discipline, and rollbackable policy versions. P1
State management Config reload requires restart. Daemon is foreground-oriented. Policy state is a JSON file. Launchd service, health probes, live config reload, crash recovery, and monotonic state transitions. P1
Realizer trust Hermes can do server-side tool use, but tool calls and reasoning are mostly opaque to the harness. Observable tool traces, explicit capability budget, prompt/version diffing, and replayable realization tests. P1
UX semantics Notch pill works and captures interactions. Approach geometry is approximate. Measured display lifecycle, no lost pending messages, accessibility, multi-monitor correctness, and tested no-overlap states. P2

Audit Fixes Applied

Correctness

  • Removed hardcoded API key from repo defaults; Settings -> Model now owns local key entry.
  • Fixed cooldown persistence across daemon ticks.
  • Made negative-feedback backoff time-bounded.
  • Made expired snoozes inactive instead of permanent blockers.
  • Made the "Later" notch action actually set a 30-minute snooze.
  • Made notch auto-dismiss respect the pending payload expiration.
  • Added local OCR privacy preflight before screenshot model calls.
  • Added harness-local screenshot masking for secret-like OCR boxes.

Data and eval

  • Preserved VLM specificity and intent signals through the event schema.
  • Included scene detail and intent signals in the realizer brief.
  • Attached outcomes and reward to matching trace rows.
  • Made hover-target reward target-aware, so hovering dismiss is soft rejection rather than positive consideration.
  • Added a privacy-safe model-call audit ledger for realizer, scene VLM, and LLM critic calls.
  • Added a typed SQLite sidecar so JSONL events can be queried without log scans.
  • Fixed the retro labeler queue so review sessions are frozen and skip moves forward instead of chasing live latest candidates.
  • Added live lab metrics for outcome capture, reward, label agreement, false interruptions, missed help, and data-readiness thresholds.
  • Made pending notch delivery idempotent with leases, so a notch crash between poll and outcome does not silently drop a message.
  • Improved replay to use live config, policy state, outcomes, and cooldown simulation.
  • URL-encoded Fisherman query tool parameters.
  • Switched session-memory timestamp parsing to UTC-safe parsing.

Risk Register

P0: Key rotation and screenshot privacy

A real-looking API key was present in the repo defaults before this audit. Even though source defaults are now blank, the old key remains in the local ~/.harness/config.toml until replaced. The harness now redacts secret-like OCR and masks sensitive screenshot text boxes locally before realizer model calls; if masking cannot be verified, screenshot attachment is suppressed.

P0: Metrics exist, but the sample is too small

The harness now reports false-interruption rate, missed-help rate, agreement, outcome capture, and reward. The remaining problem is data volume and counterfactual validity: a frontier lab would not trust these rates until there are enough labels, held-out slices, and controlled shadow-policy comparisons.

P1: Storage has a typed sidecar, but not a migration contract

The harness now mirrors JSONL into SQLite, which is enough for indexed local queries and audits. It still needs explicit migrations, backfill/recovery checks, retention policy, and a gradual read-path migration before storage can be treated as lab-grade infrastructure.

P1: Replay is still not the real system

Replay now simulates memory and cooldown better than before, but it still excludes realizer quality, critic behavior, VLM nondeterminism, user context, and delayed action. Treat it as a gate-regression tool, not a full offline product metric.

Definition Of A Completely Working Harness

Milestone Concrete bar
Dogfood-ready Daemon starts via launchd, survives app restarts, no hardcoded secrets, no update-check leakage, and 20-30 retro labels collected.
Policy-eval-ready At least 500 labeled candidates, frozen datasets, policy versioning, replay under 10 seconds for a day of candidates, and tracked precision, false-interruption rate, and missed-help rate.
Privacy-ready OCR preflight, line-box masking, and model-call audit logs exist. Remaining bar: token-level masking, explicit per-provider config, and sensitive-domain suppression.
Personalization-ready Few-shot realizer examples from accepted/rejected local labels, learned or calibrated gate features, and rollbackable policy configs.
Production-grade SQLite-backed read paths, migrations, launchd packaging, observability, crash recovery, CI on all harness paths, release notes, security review, and documented rollback path.

  • Keep the rule gate as baseline and compare learned gates against it.
  • Log model prompt version, endpoint, and image bytes for every realization.

Bottom Line

The core idea is sound: a proactive assistant should be evaluated as an intervention policy, not just as a chat prompt. This build has the right loop and enough instrumentation to start learning from real use.

The gap to "frontier lab" is disciplined measurement. The next useful work is not more prompt cleverness; it is labeled data, replay quality, storage integrity, and policy evaluation.
