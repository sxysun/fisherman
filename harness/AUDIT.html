Fisherman Harness Architecture and Gap Audit

A practical map of the proactive-presence harness as of May 25, 2026: what runs, how a screen moment becomes a ping, how feedback becomes training data, and what still separates the dogfood harness from a fully validated policy system.

Scope: harness/ plus app boundary checks Smoke tests: 72 passing in CI Approx. source/docs: 55 files, 10k lines Updated: 2026-05-25

Executive Verdict

The harness is architecturally coherent for a local research prototype. It has the right primitive loop: observe screen state, turn it into a candidate event, decide whether to ping, realize a goal-aware message, push it through a low-friction UI, capture outcomes, and audit policy behavior offline.

It is now a serious local dogfood harness, but not yet a validated policy. The engineering gaps called out in the first audit now have concrete implementations: deterministic holdouts, endpoint trust boundaries, native metrics, durable trace lifecycle, workflow-event joins, KG priors, hard-example mining, curation records, launchd packaging, capsule respawn, masked settings keys, all-Spaces capsule behavior, and SQLite-backed read paths. The remaining gap is evidence: enough labels, outcomes, and frozen evaluation slices to justify policy changes.

Best current use: dogfood on one machine with an explicit daily goal, collect labels, and use the trace data to harden the policy. Do not treat current ping quality as validated.

The harness is a binary intervention learner.

Every screen tick asks one product question: should the assistant interrupt now, or stay quiet? If it interrupts, the capsule observes what happened next. Those outcomes become weak labels, explicit labels, and eval rows for improving the next ping/not-ping policy.

Think of the harness as the lab bench around Fisherman, not Fisherman itself. Fisherman captures local context. The harness turns that context into experiments, pings, traces, and policy measurements.

One-Screen Mental Model

1

Observe

Poll Fisherman's local API, read the current app/window/OCR/screenshot, redact sensitive content, and package the moment as a typed candidate.

2

Decide

Apply hard safety gates, then let the LLM ICL policy decide ping vs. not-ping using the daily goal, recent workflow, priors, and labeled examples.

3

Learn

Deliver through the capsule, capture hover/click/dismiss/timeout signals, write a complete trace, and evaluate the policy on frozen candidate/event datasets.

How The Harness Works

Fisherman exposes local context. The harness reads from localhost. It does not need Fisherman Cloud ingest to be connected, and Fisherman does not import the harness.
The daemon makes a candidate every tick. A candidate is the atomic eval row: current app, window, OCR summary, scene tags, sensitivity state, daily goal, recent workflow, and policy state.
The policy chooses ping or not-ping. rule_v0 is the hard guardrail and fallback. llm_icl_v0 is the live policy learner: it uses in-context examples, recent trajectory, and KG-style priors.
The realizer writes the actual message. The message is not a template tied to a rigid reason code. It is generated from the daily goal, why-now rationale, compact state, and privacy-approved screen context.
The capsule records the outcome. Clicking yes/later/dismiss is recorded. Hover target, approach, timeout, and displayed acknowledgements are also captured, so ignored pings become attention-cost signal.
Eval turns behavior into policy evidence. Implicit outcomes, explicit labels, hard negatives, event labels, and frozen split reports are used to decide whether a policy change is actually better.
1. Fisherman captureFisherman daemon exposes frames, OCR, app/window metadata, status, and query endpoints on localhost.
2. Candidate synthesiscandidate.py snapshots screen context and current user prefs into a typed CandidateEvent.
3. Scene readingRules tag coarse scenes. Optional VLM overlay adds specificity, sensitivity, and intent signals.
4. Short memorySessionMemory tracks recent effective apps/scenes, switches, time on current app, and recent outcomes.
5. Context packetEventContextPacket freezes the exact policy input: current screen, raw/effective app identity, workflow run, 5-minute trajectory, examples, KG priors, optional wiki memory, and provenance.
6. Policyrule_v0 applies hard gates and fallback behavior; the default llm_icl_v0 learner chooses ping/not-ping from the frozen packet.
7. RealizerOpenAI-compatible chat endpoint receives daily goal, why-now, compact state, and optionally the current JPEG. Hermes may use its own long-term mind server-side.
8. CriticRegex critic blocks obvious secrets, long quotes, verbose output, or empty/multi-paragraph messages. LLM critic is optional.
9. Capsule + rewardSwift capsule displays the ping, captures hover/approach/click/dismiss/timeout, and writes signal-derived reward.

Fisherman -> CandidateEvent -> WorkflowEvent/Memory -> EventContextPacket -> Policy -> Realizer -> Critic -> Pending ping -> Outcome -> Reward -> Trace -> Labels/eval

Dataflow With Stored Objects

The paper's framing is event-first: recent temporal context plus user memory feeds a binary when to assist decision, and only positive decisions move to how to assist. The harness now has the same shape, but the object boundaries matter.

Screen tick CandidateEvent Atomic observation row. It carries the current screen slice, raw app metadata, scene tag, lightweight context signals, prefs, and a workflow join id.
Workflow run WorkflowEvent App/window continuity object. This is closer to the paper's event unit than a single 5-second candidate tick.
Short memory MemorySnapshot Recent effective apps, scenes, outcomes, workflow runs, switch count, time-on-app, and idle-gap state. Passed to policy, not copied into the candidate row.
Policy input EventContextPacket Frozen decision example. It materializes exactly what the policy saw: current observation, app identity, workflow context, recent outcomes, few-shot examples, KG priors, optional wiki memory, privacy state, and provenance.
When policy Decision Binary action: notch_ping or no_ping. Stores confidence, reason codes, why-now, evidence, and policy version.
Execution trace Trace Joinable lifecycle record: candidate, memory snapshot id, decision, realizer, critic, delivery, outcome, reward, and labels.
Learning rows Labels + outcomes Explicit labels and usable implicit outcomes become weak training/eval examples, hard negatives, and KG-style priors.
Join spine: candidate_id links candidates, context packets, decisions, traces, deliveries, outcomes, and labels. workflow_event_id groups many candidate ticks into one app/window episode so eval does not pretend that 30 near-identical ticks are 30 independent examples. packet_id lets audits recover the exact model-facing context for a policy decision.

Candidate Schema, In Plain English

Field What it contains now What it does not contain yet
candidate_id Stable id for this atomic observation tick. No event summary or label by itself.
ts UTC timestamp when the harness synthesized the candidate. No full preceding timeline. That lives in memory/workflow rows.
screen Raw Fisherman frontmost app, bundle id, window title, redacted OCR snippet, capture timestamp, capture gap, frame age, and sensitive-scene flag. No screenshot bytes in the candidate row, no full OCR transcript, no long-term knowledge-base summary. Effective app identity is computed in packet/workflow code, not written over the raw candidate field.
scene Rule or VLM tag: label, confidence, specificity, intent-signal booleans, and load-bearing visible text. Not a full semantic understanding of your current task. VLM output can be wrong or internally inconsistent.
context Call/battery state, minutes since last push, and placeholder user-action timing. Does not yet include mouse/keyboard activity history, browser tab stack, or semantic work objective.
user_pref Frequency, enabled intents, quiet hours, snooze, and muted intents from harness config/state. Not learned preferences except where policy state or labels indirectly influence later prompts.
workflow_event_id Pointer to the current app/window workflow event. The actual workflow event details are separate rows.

Current shape: the candidate object is intentionally thin, and that is now correct. The rich model input is persisted separately as EventContextPacket rows in ~/.harness/context_packets.jsonl and SQLite context_packets. Packets preserve raw app metadata while adding effective_app and app_metadata_mismatch when OCR menu-bar evidence shows stale metadata. The Hermes/mind bridge exists behind policy_retrieval_enabled, but serious long-term memory still needs a dedicated audited retrieval API and frozen ablations.

KG-Style Priors Are Counts, Not Training

The current KG prior builder is not fitting a model, not doing gradient descent, and not learning embeddings. It builds a local table of interpretable feature counts from explicit labels plus usable implicit outcomes.

feature = app | scene | app_scene | keyword | app_keyword target = notch_ping if label == would_help target = no_ping if label in {would_annoy, good_no_ping} p_should_ping = (positive_weight + 0.5) / (total_weight + 1.0)

The +0.5 and +1.0 are Jeffreys-style smoothing. They stop tiny samples from becoming falsely certain.

Recent prior table sample

Current local build: 111 usable examples. Top features include:

  • kw:edit: p_should_ping=0.237, n=57.0
  • app:google chrome: p_should_ping=0.257, n=52.8
  • scene:coding: p_should_ping=0.303, n=30.4
  • app_kw:google chrome|view: p_should_ping=0.184, n=27.6

This becomes soft context in the live LLM policy prompt. It should influence the policy, not override evidence from the current screen.

What this means: KG priors are a transparent memory hint. They answer, "In past local data, were moments like this usually helpful to ping on?" They are not a calibrated classifier yet. The next lab-grade step is comparing policy variants with and without these priors on frozen temporal splits.

What Is Strong

Right dependency direction

The harness depends on Fisherman through HTTP. Fisherman does not import or ship the harness as app runtime. This keeps the experimental loop replaceable.

Goal-aware intervention model

The current design avoids a brittle fixed-intent catalog. The gate emits why-now reasons and the realizer composes against the daily goal and actual visual context.

Feedback loop exists

Outcomes, hover traces, retro labels, implicit labels, score, and reward are in place. That is the right skeleton for improving intervention policy over time.

Native surface is appropriate

The floating capsule is less disruptive than a modal/chat UI, can coexist with Fisherman's notch, and can capture richer intent signals than a notification alone.

Local-first logs

All core trace artifacts live under ~/.harness. That makes dogfooding inspectable and easy to reset.

Testable seams

The modules are small enough to unit test: gate, scene overlay, reward, trace patching, delivery lifecycle, frozen eval, and snooze behavior now have 72 smoke tests.

Risk Register

P0: Key rotation and screenshot privacy

A real-looking API key was present in the repo defaults before this audit. Even though source defaults are now blank, the old key remains in the local ~/.harness/config.toml until replaced. The harness now redacts secret-like OCR and masks sensitive screenshot text boxes locally before realizer model calls; if masking cannot be verified, screenshot attachment is suppressed.

P0: Metrics exist, but the sample is too small

The harness now reports false-interruption rate, missed-help rate, precision/recall/F1, trace completeness, outcome capture, reward, first-pass shadow-policy comparisons, hard-example counts, and deterministic holdouts. The remaining problem is data volume: a frontier lab would not trust these rates until there are enough labels and held-out slices.

P1: Storage has a typed sidecar, but not a migration contract

The harness now mirrors JSONL into SQLite and dashboard/metrics/replay/score/dataset builders/shadow prefer SQLite payload rows. It has backfill and delivery-lifecycle recovery coverage, but still needs explicit migration tests across old local databases and an automatic retention/compaction policy before storage can be treated as long-lived lab infrastructure.

P1: Replay is still not the real system

Offline policy replay now simulates memory and cooldown better than before, but it still excludes realizer quality, critic behavior, VLM nondeterminism, user context, and delayed action. Treat it as a gate-regression tool, not a full offline product metric.

Bottom Line

The core idea is sound: a proactive assistant should be evaluated as an intervention policy, not just as a chat prompt. This build has the right loop and enough instrumentation to start learning from real use.

The gap to "frontier lab" is now mostly disciplined data volume and promotion policy. The next useful work is labeled data, implicit-signal calibration, retention/compaction, and native review surfaces.
